Traffic math

Write QPS = 1,000,000,000 / 86,400 ≈ 10,000 QPS (avg)
Peak     = 10,000 × 3   ≈ 30,000 QPS

Read QPS = 10,000 × 10  = 100,000 QPS (avg)
Peak     = 30,000 × 10  = 300,000 QPS

Storage math

Per like record = 32 bytes
  content_id : 8 bytes
  user_id    : 8 bytes
  timestamp  : 8 bytes
  metadata   : 8 bytes

1 year = 10^9 × 365 × 32 ≈ 10 TB

Interview trick: 365 ≈ 400, 32 ≈ 30 → 400 × 30 = 12,000 ≈ 10^4
∴ 10^9 × 10^4 = 10^13 bytes = ~10 TB ✅

Hot spot calculation

Viral post: 1M likes in 10 minutes
Write rate = 1,000,000 / 600 ≈ 1,666 writes/sec on SAME content_id 🔥
Why this matters: 1,666 writes/sec on a single DB row → lock contention → timeouts → dropped writes. Normal QPS numbers hide this. Always do the hot spot calculation separately.

API Design

POST /v1/likes Like content 
Request:  { "content_id": "post_abc", "content_type": "post", "user_id": "user_xyz" }
Response: { "success": true, "like_count": 1247, "liked_by_user": true }
Errors:   409 Already liked · 404 Not found · 429 Rate limited
DELETE /v1/likes Unlike content 
Request:  { "content_id": "post_abc", "user_id": "user_xyz" }
Response: { "success": true, "like_count": 1246, "liked_by_user": false }
Errors:   404 Never liked · 400 Bad request
GET /v1/likes/{content_id}?user_id=xyz Get count 
Response: { "content_id": "post_abc", "like_count": 1247, "liked_by_user": true }
POST /v1/likes/count/batch Batch counts 
Request:  { "content_ids": ["post_1", "post_2", ...], "user_id": "xyz" }
Response: { "results": [{ "content_id": "post_1", "like_count": 1247, "liked_by_user": true }, ...] }
Why content_type? content_id=101 could be a post, video, or comment. content_type routes validation to the correct microservice.
Why return count in write response? Avoids a second round trip — client updates UI immediately from the write response.
Why POST for batch reads? GET requests cannot carry a body. POST is acceptable when an array of IDs must be sent. This is a common interview gotcha.

Deep Dives

6.1 — Counting approaches

ApproachRead speedWrite contentionAccuracyVerdict
Real-time COUNT(*)Slow at scaleNonePerfectFails at scale
Pre-aggregated counterO(1)High for viralPerfectOK up to ~100/s
Async via KafkaO(1)NoneEventually consistentRecommended
Recommendation: Async aggregation. Kafka buffers bursts. Redis serves reads in O(1). 1-5s lag is invisible to users — they see "1.2K" anyway.

6.2 — Hot spot handling

The problem: 10,000 writes/sec on one post → all fight for same DB row lock → contention → timeouts → dropped writes. Even Redis single key at 10K INCR/sec is a bottleneck (single-threaded command execution).

Solution: Sharded counters

shard_id = hash(user_id) % 100    ← deterministic, not random

Write: INCR count:post_xyz:42         ← 100 writes/sec per shard (not 10,000)
Read:  MGET count:post_xyz:0..99 → sum all shards → cache result

Adaptive counting

rate detection (sliding window):
  INCR  rate:post_xyz:{current_minute}
  EXPIRE rate:post_xyz:{current_minute} 120

  if rate > 100/min → switch to sharded counters
  if rate < 100/min → consolidate back to single counter

transition (no data loss):
  existing count = 1,000
  seed each of 100 shards = 1,000 / 100 = 10 each
  SUM = 1,000 ✅

6.3 — Deduplication

Layer 1 — DynamoDB primary key (safety net)

try {
    INSERT INTO likes (content_id, user_id, created_at)
    return SUCCESS
} catch (DuplicateKeyException e) {
    return ALREADY_LIKED    // idempotent — not an error
}

Layer 2 — Redis cache (fast path)

EXISTS user_like:post_xyz:user_abc
  → found    → return 409 (never hits DynamoDB)
  → not found → write DynamoDB → SET Redis key, TTL 1hr
Race condition: Two concurrent requests both miss Redis cache → both attempt DynamoDB insert → one throws DuplicateKeyException. Caught gracefully. DB is the safety net.

CAP Classifications

DynamoDB Likes CP
Deduplication must be correct. If it returns stale data, a user can like the same post twice.
Kafka AP
Must keep accepting like events during partition. Refusing events would fail user operations — availability wins.
Redis AP
Stale like count is acceptable. Showing 1,247 instead of 1,248 for a few seconds is not a serious problem.
Like Service AP
Stateless service — availability is the priority. All state lives in DB and cache.
DynamoDB Counts AP
Recovery mechanism — eventual sync from Redis is fine. Not on the primary read path.
Rule of thumb: Any component doing deduplication or financial transactions → CP. Any component doing counting, caching, or buffering → AP.

Sharded counters specifically

INCR count:post_xyz:42         <- write to one shard (hash(user_id) % 100)
MGET count:post_xyz:0..99      <- read all 100 shards in one round trip
SET  count:post_xyz:total      <- cache the aggregated sum with TTL

HyperLogLog — why we rejected it

PFADD  likes:post_xyz user_abc    <- add user to set
PFCOUNT likes:post_xyz            <- estimated unique count (~0.81% error)
PFDEL  likes:post_xyz user_abc    <- DOES NOT EXIST
Fatal flaw: HyperLogLog has NO remove operation. Unlike is impossible to track. Once a user is added, they cannot be removed.
HyperLogLog limitationImpact on our system
No delete / remove operationUnlike operation becomes impossible to track
~0.81% error rateCount off by thousands at millions scale
No membership checkCannot answer "has user X liked post Y?"
Approximate onlyCannot use for deduplication

Where HyperLogLog IS useful

Good for: Unique users who viewed a post · Distinct IPs hitting an API · Unique visitor dashboards — anywhere removal is never needed and approximate is fine.
Not for: Like counts (unlikes needed) · Deduplication (needs exact membership) · Any counter requiring removal operations.

Redis failure recovery chain

Normal operation:
  All reads  -> Redis (primary)
  All counts -> INCR/DECR Redis via Aggregation Service
  Every 60s  -> sync Redis count + Kafka offset -> DynamoDB Counts

Redis goes down:
  Reads fall back to DynamoDB Counts immediately
  Likes still flow -> DynamoDB Likes + Kafka (no data loss)
  Offsets NOT committed during outage (Redis write failed)

Redis comes back:
  Step 1: Load DynamoDB Counts into Redis  (bulk recovery)
  Step 2: Replay Kafka from saved offset   (fills the gap)
  Step 3: Redis fully accurate

Why save Kafka offset alongside count sync?
  Offsets for PROCESSED events are already committed
  DynamoDB Counts may be 60s stale
  Saving offset at sync time -> replay exactly the missing window
Production defense layers: Redis AOF persistence handles most restarts — Redis recovers itself on restart. Offset + count sync to DynamoDB handles catastrophic data loss. Likes table COUNT(*) is the last resort source of truth.

Concept Cards

Sharded Counters

What: Split one counter into N counters to distribute write load across multiple keys.

When: Single counter receiving too many concurrent writes — hot spot problem.

How: shard_id = hash(user_id) % N · write to counter:id:{shard_id} · read = sum of all shards.

Why deterministic hash (not random): Same user always maps to same shard — like and unlike always cancel correctly.

Tradeoff: Reads cost N Redis calls — mitigated by caching the aggregated sum.

Sliding Window Counter

What: Track event rate over a rolling time window (not a fixed reset window).

When: Hot content detection, rate limiting, fraud detection.

How: INCR rate:post_xyz:{current_minute} · EXPIRE 120 · check if above threshold.

Why not fixed window: Fixed window misses spikes at minute boundaries. Sliding window catches all spikes continuously.

Consistent Hashing

What: Hashing technique where adding/removing nodes only remaps ~1/N keys instead of all keys.

When: Dynamically scaling sharded systems (cache clusters, DB shards).

How: Nodes placed on a ring (0–360°). Each key maps to nearest node clockwise.

Impact: Adding 1 shard remaps ~1% of keys. Without it: ~100% remapped.

Thundering Herd

What: Cache failure causes all concurrent requests to simultaneously hammer the database.

When: Redis goes down → 100M users query DynamoDB simultaneously → DB crashes.

Fix: Request coalescing — first request acquires SETNX lock, queries DB, populates cache. Others wait and read from cache.

Global Secondary Index (GSI)

What: Auto-maintained copy of data with a different partition key for an alternate query pattern.

When: Two equally important queries need different partition keys.

Example: Main table: content_id (PK) for "count likes for post". GSI: user_id (PK) for "all posts liked by user".

Cost: Extra storage + slightly slower writes.

Optimistic UI

What: Client updates UI immediately before server confirms, then reconciles on response.

When: Write latency would make UX feel sluggish — like button, follow, bookmark.

How: Fill heart + increment count locally → POST to server async → adjust UI if server returns different value.

Tradeoff: Rare brief inconsistency if server rejects — handled gracefully by reverting UI.

Write-Through Cache

What: On every write, update both DB and cache synchronously in same request flow.

When: Strong consistency required for "read your own writes" — user's own like status.

How: INSERT DynamoDB + SET Redis key, TTL 1hr — both in same like handler.

Tradeoff: Slightly slower writes (2 operations) vs read-your-own-writes guarantee.

LSM Tree (Why DynamoDB handles high writes)

What: Log Structured Merge Tree — writes go to memory first, flushed sequentially to disk.

Why fast: Sequential writes are 10× faster than random writes. No row locking.

Flow: Write → MemTable (RAM) → periodic flush → SSTable (disk, sequential).

Result: Handles 30K+ write QPS comfortably at a single node level.

Interview Pitch

Read this out loud until it flows naturally. ~60 seconds.

"The likes counting system has two core challenges — high write throughput and the hot spot problem from viral content.

For writes, I store like records in DynamoDB with a composite primary key (content_id, user_id) for deduplication, then publish events to Kafka for async count aggregation into Redis. This decouples write throughput from count updates — Kafka buffers any burst.

For reads, 300K peak QPS is served through a three-layer cache — local in-memory at 5s TTL, Redis at 10min TTL, DynamoDB counts as fallback. Most requests never reach the DB.

For hot spots, I use adaptive sharded counters — when a post's like rate exceeds 100/min, I split its Redis counter into 100 shards using hash(user_id) % 100. Each shard receives ~100 writes/sec instead of 10,000. The sum of all shards is cached so reads remain O(1).

Consistency is applied selectively — like counts are eventual (acceptable 2-3s lag), but a user's own like status uses write-through cache for immediate accuracy. Deduplication is strongly consistent via the DynamoDB primary key."
